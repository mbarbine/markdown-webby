## YYYY-MM-DD - JSON-LD XSS Prevention
**Vulnerability:** XSS vulnerability in `JsonLd` component. When embedding JSON-LD schemas, `JSON.stringify` does not escape `<` and `>` characters by default. If untrusted data is included, an attacker could break out of the script tag (e.g., `</script><script>alert(1)</script>`).
**Learning:** `dangerouslySetInnerHTML` on script tags with `JSON.stringify` requires escaping HTML characters, particularly `<`.
**Prevention:** Replace `<` with `\u003c` or use a dedicated serialization library like `serialize-javascript` when interpolating JSON directly inside a script tag.

## 2025-04-12 - Fix XSS Vulnerabilities properly with DOMPurify
**Vulnerability:** The custom Markdown to HTML conversion pipelines (`components/editor/markdown-preview.tsx` and `app/api/v1/export/route.ts`) were vulnerable to XSS. Previous attempts to manually sanitize inputs using simple string replacement for attributes and URLs were incomplete and caused formatting regressions (e.g., double encoding ampersands, breaking inline code in links).
**Learning:** Custom Markdown parsing combined with manual HTML escaping is extremely brittle, highly error-prone, and leads to regressions or bypassable security controls. Using `dangerouslySetInnerHTML` on unverified strings or manual replacements often misses edge cases (like HTML entities in `javascript:` URIs). Furthermore, when adding sanitization to a React project (especially Next.js with SSR), standard `dompurify` requires a DOM, which causes issues or fails open during Server-Side Rendering.
**Prevention:** Never rely on manual regex or string-based HTML sanitization for user input. Always use a robust, dedicated sanitization library like `isomorphic-dompurify` (which safely handles SSR/Node contexts) immediately before rendering or outputting HTML strings. This ensures safe mitigation of XSS while preserving legitimate HTML structures.
